What is CrowdSec

it is like a distributed Fail2Ban

  • parse input
  • extract important data
  • take action if pattern is found

... and much more...

  • inputs - logs from various sources
  • extract - for example source IP
  • take action - remediation, do nothing etc
  • enrich - add more metadata such as ban time, reason...

CrowdSec Remediation

Examples:
  • add bad IP to the block list referenced by iptables
  • execute a command
  • send bad IP and metadata to another service
Remediation via integration:
  • add bad IP to the block list referenced by iptables
  • execute a command
  • just expose list of blocked IP
  • send bad IP and metadata to another service
Remediation - bouncers:
  • one of the implementations of the blocking action
  • can be local - iptables/nftables/web app integration
  • can be remote - routers, CDN providers...
  • you can have multiple remediations at once
Remediation - bouncers - details:
  • CrowdSec ban is a 'decision' - what IP block, how long and why
  • when bouncer connects to the CrowdSec API it gets all decisions
  • after that bouncer streams down the incoming new decisions

MikroTik

MikroTik does pretty good networking devices

  • in this setup it is a small home lab
  • devices can be controlled over API/CLI/web
  • can be scripted with commands used by RouterOS (CLI)

Integration

Use MikroTik as bouncer in CrowdSec

  • do not code directly in the CrowdSec
  • do not call commands
  • do not use plaintext ban list or RouterOS code

Someone already did this

  • funkolab/cs-mikrotik-bouncer

great, let test this...

... but it is slow!

  • direct MikroTik API calls - fast
  • adding IP to the firewall rule - fast
  • removing IP from the firewall rule - slow

removing IP from the firewall rule - slow

  • removing single IP address took seconds
  • apparently this is an issue how MikroTik does it
  • funkolab/cs-mikrotik-bouncer instance was out of sync

The fix - do not delete IP from the list

  • if the new decision comes in, just create a new list
  • add the IPs to the list
  • update MikroTik to use specific new list (atomic update)
  • let old list to expire on their own

Features

  • run as go binary, or in container
  • 64MB per process or less
  • prometheus metrics - shiny graphs

Limitations

  • just IPv4/IPv6 on the firewall rules in MikroTik
  • very frequent changes can create many unused block lists
  • large lists vs older hardware

Future

  • return to single block list and allow IP to expire
  • IP address merging - from 10k records to 7k records
  • filters - split lists from CrowdSec and local block lists
